dploy

If you are in an appops role and want to run microservices on DC/OS in a simple and convenient way, dploy might be something for you. Essentially, dploy allows you to create, deploy and manage apps based on a collection of Marathon app specs written in JSON. To set up and manage your app across the entire lifecycle, we use convention over configuration, for example the location of resources.

Installation

From source:

$ go get github.com/mhausenblas/dploy
$ go build

To simplify the DC/OS oauth handling, you can create a SSH tunnel from your local machine to the DC/OS master like so:

$ ssh -i ~/.ssh/MYKEY core@MYMASTER -f -L 8080:localhost:8080 -N

From here on, you can use http://localhost:8080 for the Marathon URL (marathon_url) in the dploy.app file.

Workflow

dploy init

This command creates an app descriptor dploy.app.

Usage:

$ dploy init [-location=DIR]

With the following optional parameter:

  • If -location is not present, dploy.app is created in the current dir
  • If -location is present, you can specify its value DIR either absolute (e.g., /tmp/) or relative to the directory dploy is executed, for example, ../. means create dploy.app in the parent dir

Example:

$ dploy init

dploy dryrun

This command validates the app descriptor by checking if Marathon is reachable and also checks if the app spec directory is present, incl. at least one Marathon app spec.

Usage:

$ dploy dryrun

Example:

dploy run

This command launches the app as defined in the app descriptor and the app specs. It scans the specs/ directory for Marathon app specs and launches them using the Marathon API.

Usage:

$ dploy run

Example:

$ dploy run

dploy destroy

This command tears down the app by deleting the Marathon apps based on the app specs found in the specs/ directory.

Usage:

$ dploy destroy
